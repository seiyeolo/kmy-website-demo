# KMY 협회 홈페이지 리뉴얼 프로젝트

## 프로젝트 필수 공통 규칙

- 모든 코드 주석과 답변은 반드시 **한국어**로 작성
- Python 가상환경: `uv` 사용, `.venv`로 생성 (기존 폴더에 가상환경이 있다면 새로 만들지 않음)
- 시각화 가이드: `seaborn` 스타일 설정(`sns.set_theme` 등) 사용 금지
- matplotlib 한글 폰트: `koreanize-matplotlib` 필수 사용
- 새 라이브러리 추가 시 반드시 사용자에게 사전 확인

## 프론트엔드 작업 규칙

- HTML/CSS/JS 순수 바닐라로 구현 (프레임워크 없음)
- 모바일 퍼스트 반응형 디자인
- 시멘틱 HTML5 필수 (header, nav, main, section, footer)
- CSS 변수 시스템으로 일관성 유지
- 이미지는 placehold.co 플레이스홀더 사용
- 폼은 프론트엔드 유효성 검사만 구현 (백엔드 없음)
- WCAG 2.1 AA 접근성 기준 준수
- 파일 인코딩: UTF-8

## AI 코딩 규범 (Karpathy Guidelines)

AI의 일반적인 코딩 실수를 줄이기 위한 핵심 행동 지침입니다. 속도보다 신중함을 우선하며, 사소한 작업에는 판단력을 사용합니다.

### 1. 코딩 전에 생각하라

- 가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.
- 가정을 명시적으로 밝혀라. 불확실하면 질문하라.
- 여러 해석이 가능하면, 모두 제시하라 — 조용히 하나를 고르지 마라.
- 더 간단한 방법이 있으면 말하라. 필요하면 반대 의견을 제시하라.
- 뭔가 불명확하면 멈춰라. 무엇이 혼란스러운지 이름 붙여라.

### 2. 단순함 우선

- 문제를 해결하는 최소한의 코드. 추측성 코드 금지.
- 요청받은 것 이상의 기능 금지. 한 번만 쓰는 코드에 추상화 금지.
- 요청하지 않은 "유연성"이나 "설정 가능성" 금지.
- 불가능한 시나리오에 대한 에러 처리 금지.
- 200줄로 썼는데 50줄로 가능하면, 다시 써라.
- _자문: "시니어 엔지니어가 이걸 과잉 복잡하다고 하겠는가?" 그렇다면 단순화하라._

### 3. 외과적 변경

- 필요한 것만 건드려라. 자기가 만든 쓰레기만 치워라.
- 기존 코드 수정 시:
  - 옆에 있는 코드, 주석, 포맷팅을 "개선"하지 마라.
  - 고장나지 않은 것을 리팩토링하지 마라.
  - 기존 스타일에 맞춰라, 네가 다르게 할지라도.
  - 관련 없는 죽은 코드를 발견하면, 언급만 하라 — 삭제하지 마라.
- 네 변경으로 고아가 생겼을 때:
  - 네 변경이 사용하지 않게 만든 import/변수/함수는 제거하라.
  - 기존에 있던 죽은 코드는 요청받지 않는 한 제거하지 마라.
- _검증: 변경한 모든 줄이 사용자의 요청에 직접 연결되어야 한다._

### 4. 목표 기반 실행

- 성공 기준을 정의하라. 검증될 때까지 반복하라.
- 작업을 검증 가능한 목표로 변환:
  - "검증 추가해" → "잘못된 입력 테스트 작성, 그다음 통과시켜"
  - "버그 고쳐" → "버그 재현 테스트 작성, 그다음 통과시켜"
  - "X 리팩토링해" → "리팩토링 전후로 테스트가 통과하는지 확인"
- 복잡한 작업은 계획 세우기:
  - [단계] → 검증: [확인사항]
- 강한 성공 기준 = 독립적으로 루프 가능 / 약한 기준 = 끊임없는 확인 필요.

## 작업 진행 방식 (웹 리뉴얼)

## Phase 목록

```
Phase 0: 뼈대 HTML (스타일 없이 구조만)
Phase 1: CSS 변수 + 기본 레이아웃 (header, footer)
Phase 2: 히어로 + 핵심 가치 섹션
Phase 3: 나머지 메인 섹션 (교육과정, 갤러리, CTA, 공지)
Phase 4: 반응형 처리 (모바일 메뉴, 그리드 조정)
Phase 5: 인터랙션 + 애니메이션 (JS)
Phase 6: 서브 페이지 구현 (about, education, dispatch, activity, contact)
Phase 7: 최종 QA + SEO + 접근성
```

상세 내용은 `INSTRUCTIONS.md` 참조.
